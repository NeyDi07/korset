# Аудит проекта - 2026-04-24

## Область проверки

Быстрый продуктовый и технический аудит после чтения `docs/CONTEXT.md`, структуры проекта, маршрутов, экранов каталога/товара/retail, Supabase-миграций, AI API, package-скриптов и результатов проверок.

## Проверки

- `npm run build` проходит вне sandbox. Vite собрал 249 модулей, PWA service worker и выдал два предупреждения о смешанных static/dynamic imports для `supabase.js` и `offlineDB.js`.
- `npm run lint` падает: 2 ошибки в `src/screens/AuthScreen.jsx` из-за компонента `EyeBtn`, объявленного внутри render. Также есть 46 предупреждений, в основном по React hooks/effects и browser globals.
- `npm test` запускается вне sandbox, но падает: 2 теста проходят, 2 падают. Падения похожи на устаревшие тесты: слишком широкий locator `text=Körset` находит несколько элементов, а `/s/store-one` ждёт старый текст `Магазин 1`.

## Главная оценка

Körset уже не выглядит как простой прототип. У продукта есть сильная store-context идея, реальные PWA-флоу, сканер, каталог, карточка товара, профиль, offline mode, AI/RAG, Supabase RLS, retail dashboard и data moat pipeline.

Главный блокер сейчас — коммерческая готовность для магазинов. `RetailImportScreen` всё ещё в основном UI-оболочка, а импорт прайс-листа — самый короткий путь к успешному onboarding магазина.

## Сильные стороны

- Понятная ценность для покупателя: скан штрихкода даёт Fit-Check по аллергиям, халяль, диетам, нутриентам и контексту товара.
- Сильное B2B2C-направление: retail dashboard уже включает покупателей, упущенную выручку, покрытие каталога, товары, настройки и публичный профиль магазина.
- Data moat заметно продвинулся: Arbuz, Korzinavdom, NPC, USDA, R2 CDN, EAN enrichment, image pipeline.
- Offline mode хорошо подходит для in-store сценария.
- Server-side OpenAI proxy имеет JWT-проверку, rate limiting, CORS и RAG-контекст.
- Supabase RLS и миграционная дисциплина уже есть.

## Слабые стороны

- Retail import — P0, но пока не реализован дальше визуального upload-экрана.
- Lint красный.
- E2E-тестов мало, и часть тестов устарела.
- Главные экраны слишком большие: `RetailProductsScreen` около 1740 строк, `HomeScreen` около 1694, `ProfileScreen` около 1569, `ProductScreen` около 1412, `ScanScreen` около 1240.
- Inline styles и inline text встречаются часто, из-за этого сложнее держать единый дизайн и RU/KZ.
- Часть нового текста обходит централизованный i18n.
- Bundle имеет крупный chunk около 977 kB raw / 282 kB gzip; сейчас терпимо, но для mobile-first PWA нужно следить.
- БД ещё требует внимания к масштабированию `scan_events`, поисковым индексам и проверкам ownership в RPC.

## Рекомендуемый приоритет

1. Починить lint-ошибки в `AuthScreen.jsx`.
2. Реализовать настоящий flow `RetailImportScreen`: CSV/XLSX parsing, валидация колонок, preview строк, поиск дублей, upsert цен/товаров магазина, отчёт об ошибках.
3. Обновить E2E-тесты под текущий UI и актуальные данные магазинов.
4. Применить или подтвердить миграцию 011.
5. Продолжить БД-фиксы: CASCADE/unique constraints/GIN/trigram/search indexes и масштабирование `scan_events`.
6. Постепенно дробить самые большие экраны после того, как бизнес-критичный импорт станет рабочим.
