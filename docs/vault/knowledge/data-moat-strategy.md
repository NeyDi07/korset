# Data Moat — Стратегия построения базы данных Körset

> Домен: knowledge / data-moat-strategy
> Дата: 2026-04-17
> Связи: [[audit-full]] · [[product-resolution]] · [[fit-check-engine]]

---

## Текущее состояние (25/100)

База штрихкодов ЕАЭС практически пуста. Open Food Facts покрывает ~30% рынка Казахстана. Ни одного казахстанского источника данных подключено.

---

## Проблемы

### data_quality_score не используется

Поле объявлено в global_products, но нигде не применяется. Нет логики расчёта, нет фильтрации по качеству. Покупатель видит одинаковый Fit-Check для данных из надёжного источника и для AI-галлюцинации.

### external_product_cache без TTL

Закэшированные данные из OFF никогда не инвалидируются. Могут быть устаревшими годами. Магазин мог изменить состав, а кэш показывает старые данные.

### AI-данные не маркируются

AI-enriched данные показываются как факт. Покупатель не знает, что информация о халал-статусе или аллергенах сгенерирована AI и может быть ошибочной. Это юридический риск и риск репутации.

### Нет каскада источников

Нет приоритизации источников данных. Сейчас resolver берёт первый найденный результат без учёта надёжности. Нужен каскад: данные магазина → подтверждённая база КЗ → OFF → AI.

### OFF — хрупкий фундамент

Open Food Facts — волонтёрская база. Нет SLA, нет гарантии актуальности. Данные могут исчезнуть или измениться без предупреждения. Для Казахстана покрытие ~30%, данные часто неполные или устаревшие.

### AI-зависимость убивает Data Moat

Если AI может сгенерировать данные для любого штрихкода, пропадает мотивация строить реальную базу. Но AI-данные ненадёжны. Парадокс: генерация данных должна быть последним рубежом, не первым.

---

## Стратегия исправления

### 1. Реализовать data_quality_score

Взвешенный рейтинг источника данных:
- Данные магазина (owner вводил вручную): 100
- Подтверждённая казахстанская база (КазНЭУ, ЕАЭС реестр): 80
- Open Food Facts: 50
- AI-генерация: 20

Логика:
- При разрешении товара: брать запись с наивысшим data_quality_score
- При data_quality_score >= 80: полный Fit-Check без предупреждений
- При 50-79: Fit-Check с пометкой "из открытой базы, может быть неточно"
- При < 50: Fit-Check с пометкой "предположительно, проверьте упаковку"

### 2. Добавить TTL в external_product_cache

- OFF данные: TTL 30 дней
- AI-генерация: TTL 7 дней (перепроверить при следующем запросе)
- После истечения TTL: повторный запрос к источнику
- Если источник недоступен: показать кэш с пометкой "может быть устарело"

### 3. Добавить source_type и confidence в global_products

Новые поля:
- `source_type` enum: 'store_manual', 'kz_verified', 'open_food_facts', 'ai_generated'
- `confidence` float 0-1 (аналог data_quality_score но нормализованный)
- `source_updated_at` timestamptz (когда источник последний раз подтверждал данные)

### 4. Каскад resolver с приоритетом источников

Текущий каскад (resolver.js): store_products → global_products → OFF API → AI
Новый каскад:
1. store_products (данные магазина, quality=100)
2. global_products WHERE source_type='kz_verified' (quality=80)
3. global_products WHERE source_type='open_food_facts' AND source_updated_at > now()-30d (quality=50)
4. external_product_cache (OFF, с TTL проверкой)
5. AI generation (quality=20, с маркировкой)

### 5. Маркировать AI-данные в UI

- Fit-Check с AI-данными: серый/жёлтый значок "не проверено"
- Текст: "Данные определены AI. Проверьте упаковку." (RU+KZ i18n)
- Для халал-статуса: особое предупреждение (религиозная чувствительность)
- Для аллергенов: особое предупреждение (здоровье)

### 6. Исследовать казахстанские источники

Приоритетные источники:
- ЕАЭС реестр продукции (ean.kz или аналоги)
- API магазинов: Магнум, Меломан, Технодом (у крупных ритейлеров есть API/партнёрские программы)
- КазНЭУ / ДС Метрология — сертификация продукции
- Barcodelookup API (международный, но покрывает больше чем OFF)
- GS1 Kazakhstan — стандарты штрихкодирования
- Честный ЗНАК (российский аналог, может покрывать импорт в ЕАЭС)

### 7. Механика обратной связи от магазинов

- Магазин может подтвердить/исправить данные товара → source_type='store_manual', quality=100
- Подтверждённые данные всплывают выше в каскаде
- Статистика: "X товаров подтверждено владельцем" — аргумент для подписки

---

## Метрики успеха

- Покрытие штрихкодов КЗ: 30% → 70%+ (после подключения КЗ-источников)
- data_quality_score >= 80 для 50%+ товаров
- AI-данные ВСЕГДА маркированы в UI
- TTL работает на 100% кэшированных данных
- Ни один покупатель не видит AI-генерированный халал-статус без предупреждения

## SQL-изменения (миграция)

```sql
-- Добавить source_type и confidence в global_products
ALTER TABLE global_products
  ADD COLUMN IF NOT EXISTS source_type text DEFAULT 'open_food_facts'
    CHECK (source_type IN ('store_manual', 'kz_verified', 'open_food_facts', 'ai_generated')),
  ADD COLUMN IF NOT EXISTS confidence float DEFAULT 0.5
    CHECK (confidence >= 0 AND confidence <= 1),
  ADD COLUMN IF NOT EXISTS source_updated_at timestamptz;

ALTER TABLE external_product_cache
  ADD COLUMN IF NOT EXISTS ttl_expires_at timestamptz;
```
