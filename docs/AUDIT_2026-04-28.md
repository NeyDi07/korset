## 1.1 Двойная конфигурация деплоя — 🟠 HIGH

В репо одновременно лежат:
- `@/c:\projects\korset\vercel.json` (Vercel — настоящий хостинг)

**Действие:** удалить `netlify.toml`. Если когда-то потребуется fallback — вернём из git history.

## 1.2 Дублирование `/api/ai` в `vite.config.js` — 🔴 CRITICAL

`@/c:\projects\korset\vite.config.js:8-145` содержит `localApiPlugin()` — **полная вторая реализация** `/api/ai` для dev-режима. Проблемы:

1. **Нет JWT-аутентификации** — в dev любой может дёргать `/api/ai`.
2. **Нет rate-limit** — в dev можно сжечь OpenAI-бюджет в одно нажатие.
3. **Нет CORS-проверки**.
4. **Использует `SUPABASE_SERVICE_ROLE_KEY` напрямую** (`vite.config.js:43`) — если разработчик случайно запустит dev-сервер на 0.0.0.0 или публичной сети, сервисный ключ работает на любые запросы.
5. **Любая правка `api/ai.js` должна делаться в двух местах** — рассинхрон гарантирован. Уже виден: prod-версия имеет валидацию `messages` (минимальную), dev — даже её нет.
6. **Дублирование промптов** — текст системного промпта расходится между файлами.

**Действие:** в dev-режиме `vite.config.js` должен **проксировать** на `node api/ai.js` через `vite-plugin-vercel` или просто запускать `vercel dev`, а не реимплементировать.

## 1.3 Service Worker — Background Sync не работает в фоне — 🟠 HIGH

`@/c:\projects\korset\src\sw.js:89-100`:

```@/c:\projects\korset\src\sw.js:89-100
self.addEventListener('sync', (event) => {
  if (event.tag === 'sync-pending-scans') {
    event.waitUntil(flushPendingScansFromSW())
  }
})

async function flushPendingScansFromSW() {
  const allClients = await clients.matchAll({ type: 'window', includeUncontrolled: true })
  for (const client of allClients) {
    client.postMessage({ type: 'FLUSH_PENDING_SCANS' })
  }
}
```

Проблемы:
- SW при `sync`-событии **только постит сообщения открытым клиентам**. Если приложение закрыто (главный кейс «магазин-подвал, юзер вышел») — никто не получит сообщение, сканы останутся в IndexedDB.
- **Тег рассинхрон**: SW слушает `sync-pending-scans`, а в `audit-full.md:81` описан тег `flush-pending-scans`. Нужно проверить реальную регистрацию в `offlineDB.js`.
- Нет **прямой отправки на Supabase из SW** — SW должен иметь fallback: открыть IndexedDB, прочитать pending, POST в Supabase REST API напрямую.

**Риск:** заявлено «6 слоёв офлайн», но **слой 3 (Background Sync)** работает только если приложение открыто. Это противоречит маркетингу «работает в подвале».

## 1.4 Service Worker — `/api/*` кэшируется 1 час — 🟠 HIGH

`@/c:\projects\korset\src\sw.js:9-21`: NetworkFirst для всех `/api/*` с `maxAgeSeconds: 60 * 60`.

`/api/ai` детерминирован профилем юзера и товаром — но **разные товары/профили дадут одинаковый кэш-key (URL без body)**. Workbox кэширует POST-запросы по URL, не по body. Значит первый ответ AI закэшируется, и все последующие запросы вернут тот же ответ.

На самом деле Workbox не кэширует POST по умолчанию — нужно проверить, что NetworkFirst для POST просто проксирует. Если так — cache бесполезен, можно убрать. Если нет — это баг.

**Действие:** исключить `/api/*` из SW кэша полностью.

## 2.1 RLS — `stores_update_owner` позволяет менять plan/expires_at — 🔴 CRITICAL

`@/c:\projects\korset\supabase\migrations\003_enable_rls.sql:45-46`:

```sql
CREATE POLICY "stores_update_owner" ON public.stores
  FOR UPDATE TO authenticated USING (owner_id = auth.uid()) WITH CHECK (owner_id = auth.uid());
```

Политика проверяет **только что `owner_id` не меняется**. Все остальные колонки — `plan`, `expires_at`, `is_active`, `subscription_tier` — **может менять любой владелец через PostgREST**.

**Эксплуатация:** владелец магазина в DevTools запускает:
```js
supabase.from('stores').update({ plan: 'enterprise', expires_at: '2099-01-01' }).eq('id', myStoreId)
```
→ бесплатный апгрейд до enterprise.

**Это блокер монетизации**. Любая B2B SaaS-модель требует, чтобы биллинг-поля были **только service_role**.

**Фикс:** через колоночные привилегии или WHERE-проверку `OLD.plan = NEW.plan AND OLD.expires_at = NEW.expires_at` в функции-триггере.

## 2.3 RLS — `scan_events_insert_anon` спам-уязвимость — 🔴 CRITICAL

`@/c:\projects\korset\supabase\migrations\003_enable_rls.sql:96-97`:

Anon может вставлять `scan_events` с **любым `store_id`**, **любым `ean`**, **любым `found_status`**.

**Эксплуатация:** конкурент за ночь генерирует 100k фейковых сканов в магазин жертвы с `found_status='not_found'` → дашборд жертвы показывает «упущённая выручка 50 000 000 ₸». **Это убивает B2B-метрику в тенге** — основной аргумент продажи.

**Фикс:** добавить колонку `client_token uuid` (генерируется один раз на установку приложения, хранится в IndexedDB), `WITH CHECK (client_token IS NOT NULL)` + дедупликация + триггер на rate (например, не более 100 сканов в минуту с одного `client_token`).

## 2.6 Vault RAG — anon SELECT всех чанков + анонимный RPC — 🟡 MEDIUM

`@/c:\projects\korset\supabase\migrations\002_enable_vault_rag.sql:144-149`, `:160-161`:

```sql
CREATE POLICY "vault_read_anon" ... USING (true);
GRANT EXECUTE ON FUNCTION public.match_vault_chunks TO anon;
```

Любой anon (без логина) может:
1. Скачать **всю** vault knowledge base через простой SELECT.
2. Дёргать `match_vault_chunks` с любыми embeddings — это **невалидный compute** (cosine similarity по 1536-мерному вектору на N строк).

**Риски:**
- Утечка интеллектуальной собственности (Е-добавки, халал-классификация, стратегии Data Moat — это твой knowledge moat).
- Анонимный DOS через RPC: 100 запросов/сек × ANN-поиск = высокая нагрузка на pgvector.

**Фикс:**

## 2.18 `get_lost_revenue` RPC — потенциальная ошибка вычисления — 🟠 HIGH

`@/c:\projects\korset\supabase\migrations\009_store_profile_fields.sql:83-117` — посмотри логику:

```sql
SELECT COALESCE(SUM(sp.price_kzt), 0) INTO v_result FROM ... -- первая часть

SELECT v_result + COALESCE((SELECT SUM(...) FROM ...), 0) INTO v_result; -- вторая часть
```

Это работает, но **если две части пересекаются (один скан попал в обе выборки)**, сумма будет double-counted. Условия:
- Первая часть: `JOIN store_products WHERE stock_status = 'out_of_stock' AND price_kzt > 0`
- Вторая часть: `LEFT JOIN store_products ... WHERE sp.id IS NULL` (товара вообще нет в каталоге)

Условия взаимоисключающие (или товар в каталоге но out-of-stock, или его нет в каталоге). Так что dup нет. **ОК.**

Но есть другая проблема: **товара нет в каталоге** → используется `COALESCE(sp2.price_kzt, 0)` где `sp2` — это `LEFT JOIN`, который вернёт NULL. То есть **для отсутствующих товаров вклад в lost revenue всегда 0**. Это ошибка логики: если покупатель сканирует Coca-Cola которой нет в магазине, мы должны оценить упущённую выручку по средней цене аналога, а не записать 0.

**Сейчас функция недооценивает upper bound lost revenue** — продаёт магазину меньшую цифру, чем мог бы. Это отрицательно влияет на B2B-pitch.

**Фикс:** для второй части использовать median price из `global_products.specs` или другого источника.

**Фикс:** при `!currentStore` показывать `<RetailLoader />`, а не Outlet (уже есть `isStoreLoading` гард, но если cloud вернул `null` без error — exit-state не handled).

## 5.23 `UserDataContext.jsx:48-60` — двойной запрос с RLS-subquery — 🟠 HIGH

```@/c:\projects\korset\src\contexts\UserDataContext.jsx:48-60
const [favRes, scanRes] = await Promise.allSettled([
  withTimeout(supabase.from('user_favorites').select('ean').eq('user_id', internalUserId), 5000),
  withTimeout(supabase.from('scan_events').select('id', { count: 'exact', head: true }).eq('user_id', internalUserId), 5000),
])
```

На каждом mount `UserDataContext` (= каждый login + refresh) делается:
1. SELECT user_favorites WHERE user_id = ... → RLS subquery `users WHERE auth_id = auth.uid()` (без индекса, см. 2.7).
2. SELECT count(*) scan_events WHERE user_id = ... → то же.

При 50 магазинах × 10k сканов/день = большой scan_events. **`count(*)` без index — sequential scan**. На 6M строк = секунды. Поэтому стоит `withTimeout 5000ms` — **через год эти запросы будут регулярно отваливаться**.

**Фикс:**
- `idx_users_auth_id` (см. 2.7) — обязательно.
- Вместо `count: 'exact'` использовать `count: 'estimated'` (быстрее, погрешность ±5%).
- Кэшировать счётчики в `users.scan_count` колонке + триггер.

## 5.24 `ProfileContext.jsx:222` — cloud write без `await` ответа в updateProfile — 🟢 LOW
